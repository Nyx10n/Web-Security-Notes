# Preventing Cross-Site Request Forgery (CSRF)
<div align=justify>
Cross-Site Request Forgery (CSRF) attacks work by abusing a user's authenticated session. Since browsers automatically include cookies with requests, a vulnerable application may process a forged request as if it came from the legitimate user.

Unlike XSS, CSRF is generally easier to prevent because there are well-established security mechanisms designed specifically to stop it.


## 1. CSRF Tokens (Most Effective)

A CSRF token is a unique, unpredictable value generated by the server and included in forms or requests. When a request is submitted, the server verifies that the token matches the one stored for the user's session. If the token is missing or invalid, the request is rejected. Because attackers cannot guess or read the victim's CSRF token, they cannot create valid forged requests.


## 2. SameSite Cookies

The `SameSite` cookie attribute tells browsers when cookies should be included with requests.

There are three values:

- `Strict` - Cookies are only sent when navigating within the same website.
- `Lax` - Cookies are sent for most same-site requests and some top-level navigation.
- `None` - Cookies are sent with all requests but must also be marked as `Secure`.

Using `SameSite=Lax` or `SameSite=Strict` helps prevent browsers from automatically including session cookies in cross-site requests.


## 3. Verify the Origin or Referer Header

The server can examine the `Origin` or `Referer` HTTP headers to determine where a request originated. If the request comes from an untrusted domain, it can be rejected. Although useful, these headers should be used as an additional security layer rather than the only protection.


## 4. Re-authentication for Sensitive Actions

Critical operations such as:

- Changing a password
- Updating an email address
- Deleting an account
- Performing financial transactions

should require the user to re-enter their password or complete another verification step.

Even if a forged request reaches the server, the attacker cannot complete the action without the additional authentication.


## 5. Multi-Factor Authentication (MFA)

Multi-Factor Authentication does not directly prevent CSRF attacks, but it reduces the impact of attacks involving sensitive account changes. For example, changing account credentials or disabling security settings may require a second authentication factor.


## 6. Avoid Using GET Requests for State-Changing Actions

Requests that modify application data should use POST, PUT, PATCH, or DELETE instead of GET. Sensitive actions should never be performed simply by visiting a URL. While this does not completely prevent CSRF, it helps reduce accidental exploitation and follows good security practices.


## Why Is CSRF Easier to Prevent Than XSS?

CSRF has a relatively well-defined attack path.

The attack depends on the browser automatically sending authentication credentials with a forged request. Once developers verify that the request genuinely originated from the legitimate application using techniques such as CSRF tokens and SameSite cookies, the attack is usually prevented.

XSS, on the other hand, has many different injection contexts and execution paths, making it significantly more difficult to eliminate completely.

# Summary

Modern web applications should use multiple layers of protection against CSRF, including:

- CSRF Tokens
- SameSite Cookies
- Origin and Referer validation
- Re-authentication for sensitive actions
- Multi-Factor Authentication (MFA)
- Proper use of HTTP methods

Using these techniques together greatly reduces the risk of successful Cross-Site Request Forgery attacks.
